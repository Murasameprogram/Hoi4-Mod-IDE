# HOI4 Mod IDE 主程序接口规范

**文档版本**：1.0  
**适用主程序版本**：≥ 1.0.0  

本规范详细定义 HOI4 Mod IDE 主程序与插件之间的所有标准化通信接口。插件开发者应遵循本文档以实现与主程序及其他插件的正确交互。

所有通信均基于 JSON 消息格式，通过 `HostMessaging` 类（由 `HostComms.dll` 提供）进行。插件在初始化时获得 `HostMessaging` 实例，通过它发起请求或响应广播。

---

## 1. 通信基础

### 1.1 消息格式

```json
{
  "id": "可选-UUID",
  "from": "发送方标识（主程序为 \"host\"，插件为插件ID）",
  "to": "接收方标识",
  "action": "操作名称",
  "payload": { ... },
  "timestamp": "ISO 8601 时间戳",
  "version": "1.0"
}
```

### 1.2 响应包装

对于请求-响应模式，响应中的 `payload` 采用统一包装：

```json
{
  "success": true,
  "data": { ... }                     // 成功时返回的数据
}
```

或

```json
{
  "success": false,
  "error": {
    "code": "ERROR_CODE",
    "message": "人类可读的错误描述"
  }
}
```

插件端通过 `SendToHostAsync<TReq, TRes>` 调用，成功时直接返回 `data` 部分，失败时抛出异常，异常消息包含错误码和描述。

## 2. 插件发起的服务请求（Request）

### 2.1 获取当前项目信息

**动作名** `getProjectInfo`

获取当前在 IDE 中打开的模组项目的元数据。若未打开任何项目，返回 `null`。

- **请求参数**：无（可传递 `null` 或空对象）
- **响应数据**（`data` 对象）：

| 字段 | 类型 | 必选 | 说明 |
|------|------|------|------|
| `name` | string | 是 | 模组名称 |
| `id` | string | 是 | 模组唯一标识符 |
| `version` | string | 是 | 版本号 |
| `path` | string | 是 | 模组文件夹绝对路径 |
| `author` | string | 否 | 作者 |
| `description` | string | 否 | 描述 |

**使用场景**：插件需要访问项目文件、显示项目信息或根据项目路径执行操作时调用。

---

### 2.2 记录日志

**动作名** `log`

将日志消息写入主程序的日志系统（文件、控制台等）。

- **请求参数**：

| 字段 | 类型 | 必选 | 说明 |
|------|------|------|------|
| `level` | string | 是 | 日志级别：`"Information"`、`"Warning"`、`"Error"` |
| `message` | string | 是 | 日志内容 |
| `exception` | string | 否 | 异常堆栈或错误详情（仅当 `level` 为 `"Error"` 时建议提供） |

- **响应**：无（单向消息）

**使用场景**：插件在关键操作、错误捕获或调试信息处调用，便于问题追踪。

---

### 2.3 导航到指定页面

**动作名** `navigate`

请求主程序切换到指定的内置页面。

- **请求参数**：

| 字段 | 类型 | 必选 | 说明 |
|------|------|------|------|
| `page` | string | 是 | 页面标识符，如 `"Settings"`、`"About"`、`"ModInfo"`、`"Plugins"` |
| `parameter` | string | 否 | 可选参数，如选项卡名称 `"tab=general"` |

- **响应**：无（单向消息）

**使用场景**：插件提供快捷入口，引导用户至相关功能页面。

---

### 2.4 打开新窗口

**动作名** `openWindow`

请求主程序打开一个新窗口，可显示插件视图或内置对话框。

- **请求参数**：

| 字段 | 类型 | 必选 | 说明 |
|------|------|------|------|
| `title` | string | 是 | 窗口标题 |
| `contentType` | string | 是 | 内容类型：`"plugin"`（插件视图）或 `"dialog"`（内置对话框） |
| `pluginId` | string | 当 `contentType` 为 `"plugin"` 时必选 | 插件 ID |
| `dialogName` | string | 当 `contentType` 为 `"dialog"` 时必选 | 内置对话框名称，如 `"NewModDialog"` |
| `width` | int | 否 | 窗口宽度（默认 800） |
| `height` | int | 否 | 窗口高度（默认 600） |
| `owner` | string | 否 | 所有者窗口标识（通常为 `"main"`） |

- **响应**：无（单向消息）

**使用场景**：插件需要在独立窗口中展示复杂 UI 或复用主程序的标准对话框。

---

### 2.5 读取全局配置

**动作名** `getConfig`

读取主程序的全局配置项（如主题、语言等）。配置值可为任意 JSON 类型。

- **请求参数**：

| 字段 | 类型 | 必选 | 说明 |
|------|------|------|------|
| `key` | string | 是 | 配置键名 |

- **响应数据**：任意 JSON 值，若键不存在返回 `null`。

**使用场景**：插件需要获取用户偏好或全局设置。

---

### 2.6 写入插件配置

**动作名** `setConfig`

写入插件自身的配置项，主程序按插件 ID 隔离存储。

- **请求参数**：

| 字段 | 类型 | 必选 | 说明 |
|------|------|------|------|
| `key` | string | 是 | 配置键名 |
| `value` | any | 是 | 任意 JSON 值 |

- **响应**：无（单向消息）

**使用场景**：插件持久化自身设置（如用户选项、窗口位置）。

---

### 2.7 获取当前主题

**动作名** `getTheme`

获取主程序当前主题模式。

- **请求参数**：无
- **响应数据**：字符串，`"Light"` 或 `"Dark"`。

**使用场景**：插件初始化或动态调整 UI 时调用，确保界面与主程序风格一致。

---

## 3. 主程序广播的事件通知（Event）

主程序会在特定时机向所有插件广播事件。插件通过订阅 `HostMessaging.MessageReceived` 事件或使用 `Subscribe<T>` 方法接收。

所有广播均为单向消息，无需响应。

### 3.1 主题变更

**动作名** `themeChanged`

- **广播时机**：用户切换主题后立即发送。
- **Payload**：`{ "theme": "Light"|"Dark" }`

### 3.2 项目打开

**动作名** `projectOpened`

- **广播时机**：模组项目成功打开并加载完成后。
- **Payload**：`{ "name": string, "id": string, "path": string }`

### 3.3 项目关闭

**动作名** `projectClosed`

- **广播时机**：项目关闭前（项目信息仍可访问）。
- **Payload**：`{ "id": string }`

### 3.4 项目保存

**动作名** `projectSaved`

- **广播时机**：项目保存成功后。
- **Payload**：`{ "id": string }`

### 3.5 插件即将禁用

**动作名** `pluginDisabling`

- **广播时机**：插件被禁用或主程序关闭前。
- **Payload**：无（可传递 `null`）

### 3.6 插件即将卸载

**动作名** `pluginUnloading`

- **广播时机**：主程序关闭前，插件即将被卸载。
- **Payload**：无

### 3.7 插件状态查询（请求-响应）

**动作名** `getPluginStatus`

此事件为请求-响应模式，主程序主动查询插件状态。插件需通过 `HostMessaging` 发送响应。

- **请求 Payload**：无
- **响应要求**：返回一个包含以下字段的对象：
  - `version` (string)：插件版本
  - `isBusy` (bool)：是否在执行耗时操作
  - `customStatus` (string, optional)：自定义状态文本

### 3.8 其他插件加载

**动作名** `pluginLoaded`

- **广播时机**：任何其他插件加载完成后。
- **Payload**：`{ "pluginId": string, "name": string }`

### 3.9 其他插件卸载

**动作名** `pluginUnloaded`

- **广播时机**：其他插件卸载后。
- **Payload**：`{ "pluginId": string }`

## 4. 错误码参考

当请求-响应服务失败时，异常消息中包含以下错误码：

| 错误码 | 说明 |
|--------|------|
| `INVALID_REQUEST` | 请求参数格式错误或缺少必要字段 |
| `ACTION_NOT_FOUND` | 指定的动作名不存在 |
| `PROJECT_NOT_OPEN` | 当前没有打开的项目 |
| `PAGE_NOT_FOUND` | 导航目标页面不存在 |
| `DIALOG_NOT_FOUND` | 指定的内置对话框不存在 |
| `CONFIG_KEY_NOT_FOUND` | 配置键不存在 |
| `INTERNAL_ERROR` | 主程序内部错误 |

## 5. 最佳实践与注意事项

### 5.1 错误处理
- 所有 `SendToHostAsync` 调用应置于 `try-catch` 中，根据错误码执行降级逻辑（如显示用户友好提示）。
- 单向消息也可能因序列化等问题抛出异常，建议同样捕获。

### 5.2 线程安全
- `HostMessaging` 的方法是线程安全的，可从任意线程调用。
- 事件回调可能在非 UI 线程触发，若需更新 UI，使用 `Dispatcher.Invoke` 或 `Dispatcher.BeginInvoke`。

### 5.3 资源管理
- 在 `pluginDisabling` 或 `pluginUnloading` 事件中执行清理：
  - 注销通过 `PluginMessaging` 注册的服务。
  - 取消异步操作，释放非托管资源。
  - 移除事件订阅，防止内存泄漏。
- 插件卸载后切勿继续使用 `HostMessaging` 或 `PluginMessaging` 实例。

### 5.4 性能考量
- 避免高频调用请求-响应接口（如每帧调用）。考虑缓存数据或使用广播更新。
- 广播事件会送达所有插件，处理函数应快速返回，避免耗时操作。
- 大体积 payload（如数 MB）应避免在常规通信中使用；必要时采用分块或文件共享。

### 5.5 版本兼容性
- 主程序将保持向后兼容：现有动作名、参数和响应结构在次要版本升级中不会破坏性变更。
- 插件应在 `plugin.json` 中声明 `minHostVersion`，主程序在加载时会检查，不满足则跳过。

### 5.6 调试建议
- 在插件开发时，可在 `PluginCommunication` 中添加日志记录所有收发消息（注意隐私）。
- 使用主程序输出窗口查看日志；插件记录的 `log` 消息也会出现。
- 插件加载失败时，检查插件文件夹是否包含全部必需 DLL（尤其是 `HostComms.dll`、`PluginComms.dll`），并确认 `plugin.json` 的 `entryPoint` 与实际文件名一致。

### 5.7 安全性
- 主程序强制重写所有消息的 `from` 字段为真实发送方 ID，杜绝身份伪造。
- 注册的服务默认对所有插件公开；如需权限控制，可在服务处理函数中检查调用方 ID（通过 `PluginMessageEventArgs.Message.From`）。
